## 一、跳表 SkipList

### 1. 核心定义
**有序链表 + 多层随机索引**
- **痛点解决**：普通有序链表查找需遍历，时间复杂度 $O(n)$。
- **优化思路**：在链表基础上增加多级索引，通过“空间换时间”，将查找效率提升至接近平衡树的 $O(\log n)$。

### 2. 工作原理
1. **底层结构**：最底层是包含所有元素的完整有序单链表。
2. **索引构建**：
   - 向上逐层建立稀疏索引层。
   - 上层节点是下层节点的子集，层级越高，节点越少，跨度越大。
3. **查找过程**：
   - 从**最高层索引**的头节点开始。
   - 向右比较，若下一个节点值大于目标值或为空，则**向下移动一层**。
   - 重复上述过程，直到在最底层找到目标或确定不存在。
4. **插入/删除**：
   - 先查找到插入位置。
   - **随机化策略**：通过抛硬币（随机数生成）决定新节点是否晋升到上一层索引，保证索引层的平衡性（无需像红黑树那样复杂旋转）。

### 3. 性能分析
| 指标 | 复杂度/说明 |
| :--- | :--- |
| **平均时间** | 查找、插入、删除均为 $O(\log n)$ |
| **最坏时间** | $O(n)$（极端情况下退化为单链表，但概率极低） |
| **空间复杂度** | $O(n)$（额外存储索引指针，通常约为原数据的 1/2 到 1 倍） |

### 4. 对比与工程应用
- **VS 红黑树/AVL树**：
  - **优势**：实现简单，无复杂的旋转平衡操作；范围查询更高效（链表天然有序）；并发控制更容易（可针对每段链表加锁）。
  - **劣势**：空间占用略高。
- **经典应用**：
  - **Redis ZSet（有序集合）**：当成员数量较多或成员字符串较长时，底层采用跳表实现。
  - **LevelDB/RocksDB**：内存中的 MemTable 使用跳表结构。

### 💡 记忆口诀
> 有序链表加索引，分层跳跃快查找。  
> 随机升降保平衡，无须旋转易并发。  
> Redis ZSet 核心构，空间换时效率高。

## 二、布隆过滤器 Bloom Filter

### 1. 核心定义
**基于位数组与多哈希函数的概率型数据结构**
- **痛点解决**：在海量数据场景下，传统 HashSet/HashMap 占用内存过大，且无法高效进行“存在性预判”。
- **优化思路**：牺牲一定的准确性（允许误判），换取极低的空间复杂度和极高的查询效率。

### 2. 工作原理
1. **初始化**：创建一个长度为 $m$ 的二进制位数组（Bit Array），所有位初始化为 0。
2. **插入元素**：
   - 使用 $k$ 个独立的哈希函数对元素进行计算，得到 $k$ 个索引位置。
   - 将位数组中这 $k$ 个位置的值置为 1。
3. **查询元素**：
   - 同样使用 $k$ 个哈希函数计算索引。
   - **判定逻辑**：
     - 若任意一个索引位置的值为 0 $\rightarrow$ **元素一定不存在**。
     - 若所有索引位置的值均为 1 $\rightarrow$ **元素可能存在**（存在误判概率）。

### 3. 关键特性
| 特性 | 说明 |
| :--- | :--- |
| **假阴性（False Negative）** | **不可能发生**。如果布隆过滤器说元素不存在，那它一定不存在。 |
| **假阳性（False Positive）** | **可能发生**。如果布隆过滤器说元素存在，它可能实际上不存在（哈希冲突导致）。 |
| **不支持删除** | 普通布隆过滤器无法安全删除元素（因为多个元素可能共享同一个位，清零会影响其他元素）。<br>*注：计数布隆过滤器（Counting Bloom Filter）可通过计数器实现删除，但空间开销增大。* |
| **空间效率** | 极高，通常每个元素仅需几比特到十几比特。 |

### 4. 误差控制
- **误判率影响因素**：
  1. 位数组长度 $m$：越大，误判率越低。
  2. 哈希函数个数 $k$：存在最优值，过多或过少都会增加误判率。
  3. 已插入元素数量 $n$：越多，位数组越满，误判率越高。
- **最佳实践**：根据预期的元素数量 $n$ 和可接受的误判率 $p$，反向计算所需的位数组大小 $m$ 和哈希函数个数 $k$。

### 5. 应用场景
- **缓存穿透防护**：在访问数据库前，先通过布隆过滤器判断键是否存在。若不存在，直接返回，避免请求打到数据库。
- **大数据去重**：如爬虫系统中判断 URL 是否已抓取，垃圾邮件过滤中判断发件人是否在黑名单。
- **分布式系统**：HBase、Cassandra 等使用布隆过滤器快速判断数据文件（SSTable）中是否包含某行数据，减少磁盘 IO。

### 💡 记忆口诀
> 位图哈希组合拳，海量数据判存在。  
> 说不存在必没有，说存在时可能有。  
> 空间极小速度快，缓存穿透它来挡。  
> 切记不可随意删，误判概率需考量。

## 三、单调栈 (Monotonic Stack)

### 1. 核心定义
**维护单调性（递增/递减）的栈结构**
- **痛点解决**：暴力求解“下一个更大/更小元素”需要 $O(n^2)$。
- **优化思路**：利用栈的 LIFO 特性，在遍历过程中动态维护一个单调序列，将查找最近极值的时间复杂度降为 $O(1)$，整体降至 $O(n)$。

### 2. 工作原理
1. **单调递减栈**：栈底到栈顶元素逐渐变小。
   - **作用**：寻找右边第一个**比当前大**的元素。
   - **逻辑**：当新元素 > 栈顶时，栈顶元素的“右边第一个更大元素”就是新元素，弹出栈顶并记录结果。
2. **单调递增栈**：栈底到栈顶元素逐渐变大。
   - **作用**：寻找右边第一个**比当前小**的元素。
   - **逻辑**：当新元素 < 栈顶时，栈顶元素的“右边第一个更小元素”就是新元素，弹出栈顶并记录结果。

### 3. 经典场景（面试高频）
- **Next Greater Element**：下一个更大元素 I/II
- **Daily Temperatures**：每日温度（等待天数）
- **Trapping Rain Water**：接雨水（按列计算或按行计算）
- **Largest Rectangle in Histogram**：柱状图中最大的矩形

### 4. Java 通用模板（以“下一个更大元素”为例）
> **注意**：通常存储**索引**而非值，以便计算距离或处理重复值。
``` java
import java.util.Stack;
import java.util.Arrays;

public class MonoStackExample {
    public static int[] nextGreaterElement(int[] nums) {
        int n = nums.length;
        int[] res = new int[n];
        Arrays.fill(res, -1); // 默认值为 -1，表示不存在
        Stack<Integer> stack = new Stack<>(); // 存索引，保持单调递减

        for (int i = 0; i < n; i++) {
            // 当前元素大于栈顶索引对应的元素 -> 找到栈顶元素的“下一个更大”
            while (!stack.isEmpty() && nums[i] > nums[stack.peek()]) {
                int index = stack.pop();
                res[index] = nums[i]; // 记录结果
            }
            stack.push(i);
        }
        return res;
    }

    public static void main(String[] args) {
        int[] nums = {2, 1, 2, 4, 3};
        System.out.println(Arrays.toString(nextGreaterElement(nums))); 
        // 输出: [4, 2, 4, -1, -1]
    }
}
```
### 5. 性能分析
- **时间复杂度**：$O(n)$。每个元素最多入栈一次、出栈一次。
- **空间复杂度**：$O(n)$。栈的空间开销。

## 四、单调队列 (Monotonic Queue)

### 1. 核心定义
**维护单调性（递增/递减）的双端队列 (Deque)**
- **痛点解决**：滑动窗口中求最值，暴力法每次遍历窗口需 $O(k)$，总复杂度 $O(nk)$。
- **优化思路**：队头始终是当前窗口的最值。通过剔除“不可能成为最值”的元素，将均摊复杂度降为 $O(1)$。

### 2. 工作原理
1. **维护单调递减队列**（求窗口最大值）：
   - **入队**：新元素从队尾进入，若队尾元素 < 新元素，则队尾出队（因为新元素更晚过期且更大，旧元素永无出头之日）。
   - **出队**：若队头索引超出窗口范围（`index <= i - k`），则队头出队。
   - **取值**：队头即为当前窗口最大值。

### 3. 经典场景
- **Sliding Window Maximum**：滑动窗口最大值 (LeetCode 239)
- **Shortest Subarray with Sum at Least K**：和至少为 K 的最短子数组
- **最大宽度坡**等区间极值问题

### 4. Java 通用模板（滑动窗口最大值）
```  java
import java.util.Deque;
import java.util.LinkedList;
import java.util.Arrays;

public class MonoQueueExample {
    public static int[] maxSlidingWindow(int[] nums, int k) {
        if (nums == null || nums.length == 0) return new int[0];
        
        int n = nums.length;
        int[] res = new int[n - k + 1];
        // 存索引，保持队列对应的值单调递减
        Deque<Integer> deque = new LinkedList<>();

        for (int i = 0; i < n; i++) {
            // 1. 移除窗口外元素（队头过期）
            if (!deque.isEmpty() && deque.peekFirst() <= i - k) {
                deque.pollFirst();
            }
            
            // 2. 维护单调性（移除队尾比当前小的元素）
            while (!deque.isEmpty() && nums[deque.peekLast()] < nums[i]) {
                deque.pollLast();
            }
            
            // 3. 当前元素入队
            deque.offerLast(i);
            
            // 4. 记录结果（窗口形成后）
            if (i >= k - 1) {
                res[i - k + 1] = nums[deque.peekFirst()];
            }
        }
        return res;
    }

    public static void main(String[] args) {
        int[] nums = {1, 3, -1, -3, 5, 3, 6, 7};
        int k = 3;
        System.out.println(Arrays.toString(maxSlidingWindow(nums, k)));
        // 输出: [3, 3, 5, 5, 6, 7]
    }
}
```

### 5. 性能分析
- **时间复杂度**：$O(n)$。每个元素最多入队一次、出队一次。
- **空间复杂度**：$O(k)$。队列中最多存储 $k$ 个元素。

# 四者终极对比速记

| 结构 | 核心作用 | 本质 | 典型应用 | 关键点 |
| :--- | :--- | :--- | :--- | :--- |
| **跳表** | 有序数据快速查改 | 多层索引有序链表 | Redis ZSet | 空间换时间，随机层级 |
| **布隆过滤器** | 海量元素存在性预判 | 位数组 + 多哈希 | 防缓存穿透 | 判无必无，判有可能有 |
| **单调栈** | 找左右最近更大/小元素 | 栈 + 单调性维护 | 接雨水、下一个更大元素 | 存索引，破坏单调即弹出 |
| **单调队列** | 滑动窗口最值 | 双端队列 + 单调性 | 滑动窗口最大值 | 队头最值，队尾维护单调 |
